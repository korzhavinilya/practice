### Стриминг возможен и в HTTP/1.1, и в HTTP/2, и в HTTP/3

#### HTTP/1.1 и "Chunked Transfer Encoding"

В HTTP/1.1 стриминг реализуется через заголовок Transfer-Encoding: chunked.

- Как это работает: Обычно сервер присылает заголовок Content-Length, чтобы браузер знал, сколько байт ждать. Но при стриминге сервер не знает финального размера страницы (потому что он еще ждет данные из БД).
- Вместо этого он присылает Transfer-Encoding: chunked.
- Данные отправляются кусками (чанками). Каждый кусок начинается с размера данных в шестнадцатеричном формате, затем сами данные. Запрос завершается чанком нулевого размера.
- Именно это происходит у тебя сейчас: Твой next dev сервер работает по HTTP/1.1 и просто "выплевывает" чанки в открытую трубу.

#### HTTP/2 и HTTP/3 (Multiplexing)

В современных протоколах (H2/H3) всё работает еще круче.

- Вместо текстовых чанков используются бинарные фреймы.
- Мультиплексирование: HTTP/2 позволяет передавать несколько "потоков" (streams) внутри одного TCP-соединения.
- Для Next.js это означает, что стриминг страницы, загрузка картинок и шрифтов могут идти одновременно, не блокируя друг друга (решается проблема Head-of-line blocking).
В современных протоколах (H2/H3) всё работает еще круче.

#### Почему на локалхосте ты видишь HTTP/1.1?

Большинство локальных серверов (включая стандартный запуск next dev) по умолчанию используют HTTP/1.1, потому что:

- Отсутствие SSL: HTTP/2 в браузерах работает только через HTTPS. Настроить локальный сертификат — лишняя морока для базового инструмента.
- Простота: Для разработки возможностей 1.1 (включая стриминг) хватает за глаза.

На заметку для собеса: Если тебя спросят, как оптимизировать доставку RSC в продакшене, смело говори: "Убедиться, что сервер/CDN поддерживает HTTP/2 или HTTP/3, чтобы стриминг чанков не конкурировал за соединение с тяжелыми ассетами".

#### Бонусный вопрос: А что если между сервером и клиентом стоит Proxy? (Nginx/Cloudflare)

Это классический подвох на интервью. Если у тебя старый Nginx с дефолтными конфигами или какой-то хитрый прокси-слой, он может попытаться сбуферизировать ответ. То есть он дождется, пока Next.js отдаст всё, и только потом вывалит это клиенту. В таком случае твой Suspense на клиенте "сломается": ты будешь 3 секунды смотреть на белый экран, а потом страница появится целиком.

Решение: Для стриминга прокси должен поддерживать proxy_buffering off.

### Виды рендеринга

В Next.js 13+ (и актуальной 15-й версии) магия происходит через расширенный fetch

#### Server Side Rendering (Dynamic Rendering)

Данные запрашиваются при каждом визите.

#### Static Site Generation (SSG)

Данные кешируются на этапе сборки.

#### Incremental Static Regeneration (ISR)

Обновляем статику в фоне раз в N секунд.

#### Прямой доступ к БД и Streaming

Поскольку RSC не попадает в бандл клиента, мы можем писать SQL прямо в теле функции. А с помощью Suspense мы реализуем Streaming — отдаем оболочку страницы сразу, а тяжелые данные «досылаем» по мере готовности.

RSC Payload: Это не HTML. Это дерево, где указаны пропсы для клиентских компонентов и результат рендера серверных. Клиентский React (библиотека) парсит этот поток и «вклеивает» в DOM. Это позволяет сохранять стейт на клиенте при навигации, даже если данные пришли с сервера.

"Waterfall". Если ты делаешь await двух разных запросов в одном компоненте последовательно, это плохо. Используй Promise.all() или разноси их по разным Suspense, чтобы они стримились параллельно.

#### Client Side Rendering (CSR)

Client Components: Чтобы использовать useState, useEffect или браузерные API, добавь 'use client' в начало файла. Помни: клиентский компонент рендерится и на сервере (пре-рендер в HTML), и на клиенте. Сервер выполняет тело функции твоего компонента, чтобы превратить JSX в HTML-строку, без выполнения useEffect.

Вот как это работает поэтапно:

- На сервере (Pre-rendering):
  - Next.js берет твой 'use client' компонент.
  - Он выполняет саму функцию MyComponent().
  - НО: Он полностью игнорирует useEffect и useLayoutEffect. В мире Node.js нет жизненного цикла «монтирования в DOM», поэтому эти хуки там просто не запускаются.
  - Сервер видит твой JSX, превращает его в HTML и отправляет в браузер.
- На клиенте (Hydration):
  - Браузер получает HTML и показывает его (LCP).
  - Затем загружается JS-бандл React.
  - React «оживляет» (гидрирует) этот HTML.
  - Вот только сейчас срабатывает useEffect.

На интервью могут спросить: «Почему нельзя обращаться к window или localStorage прямо в теле клиентского компонента, даже если там стоит 'use client'?»

Правильный ответ: Потому что этот код упадет с ошибкой на сервере во время пре-рендеринга (window is not defined). Любые обращения к браузерным API должны быть либо внутри useEffect, либо обернуты в проверку if (typeof window !== 'undefined').

'use client' — это не команда «рендерить только в браузере». Это команда «включить этот компонент в JS-бандл и разрешить ему интерактивность». Но первичный HTML для него всё равно подготовит сервер (если ты не отключил это принудительно через ssr: false).

### Кэширование и Инвалидация

В Next.js кэширование — это не "приятное дополнение", а поведение по умолчанию. В режиме Dev всё кажется динамичным, но в Production проект превращается в "застывшую статую", если ты не управляешь инвалидацией.

#### Инструменты инвалидации

- `revalidatePath` (Грубый инструмент)
  Работает по URL-пути. Говорит серверу: "Всё, что связано с этим адресом, протухло. Пересчитай".
  Где использовать: Когда мутация влияет на всю страницу или несколько компонентов сразу.
  Синтаксис:
  ```
  revalidatePath('/shop/products'); // Конкретная страница
  revalidatePath('/shop', 'layout'); // Страница и все вложенные (подкатегории)
  ```
- `revalidateTag` (Ювелирный инструмент)
  Позволяет пометить конкретные вызовы данных тегом и сбрасывать только их, не трогая остальную страницу.
  Где использовать: Когда на одной странице много разных источников данных (профиль, лента, реклама), и ты не хочешь перерендеривать всё подряд.
  Синтаксис:
  ```
  // 1. Привязываем тег к fetch
  const res = await fetch('...', { next: { tags: ['products'] } });

  // 2. В Action сбрасываем по тегу
  revalidateTag('products');
  ```

| Static Rendering | Страницы перерендериваются при каждом запросе.  | Страницы рендерятся один раз при билде. Без ревалидации они не обновятся.                                         |
| Router Cache     | Сбрасывается часто.                             | Держится в памяти браузера до 5 минут (для динамики) или 30 секунд.                                               |
| Router Cache     | Client-side                                     | Кэш в браузере. Когда ты ходишь по , Next хранит уже посещенные страницы в памяти, чтобы не запрашивать их снова. |


Твой главный риск: Забыть revalidatePath. В итоге ты добавил товар в БД, а пользователь (и ты сам в проде) видишь старый список.

### Что такое React Compiler (React Forget)?

Это билд-тайм инструмент, который выполняет глубокий статический анализ кода. Он автоматически внедряет мемоизацию на уровне графа зависимостей, используя внутренний кэш React 19 (useMemoCache (это низкоуровневый хук, который не предназначен для ручного использования)). Это избавляет разработчика от ручного управления useMemo и useCallback, гарантируя максимальную производительность при условии соблюдения чистоты функций (Rules of React).

Как он работает «под капотом»?

- Компилятор парсит твой код и проверяет его на «чистоту». Он ищет нарушения правил React (например, мутации пропсов или переменных, созданных во время рендера). Bailout (Откат): Если компилятор видит, что компонент написан «грязно» и его опасно оптимизировать, он просто пропускает его и оставляет как есть. Приложение не ломается, но и магии не происходит.
- Построение графа зависимостей. Он анализирует поток данных. Если ты создаешь объект или массив, компилятор видит, от каких переменных (пропсов или стейта) этот объект зависит.
- Трансформация (Memoization Cache). Вместо того чтобы добавлять привычные нам useMemo или useCallback, компилятор переписывает код, используя специальный внутренний механизм React 19 — useMemoCache (это низкоуровневый массив для хранения вычисленных значений).

Твой исходный код:

```
function Product({ items, filter }) {
  // Ты не написал useMemo, и при каждом рендере
  filterItems создается заново
  const filtered = items.filter(i => i.type === filter);
  
  return <List data={filtered} />;
}
```

Что делает Компилятор (упрощенно):

```
function Product({ items, filter }) {
  const $ = useMemoCache(4); // Резервирует 4 слота в кеше

  let filtered;
  // Проверяет: изменились ли items или filter в кеше?
  if ($[0] !== items || $[1] !== filter) {
    filtered = items.filter(i => i.type === filter);
    $[0] = items;
    $[1] = filter;
    $[2] = filtered; // Сохраняет результат
  } else {
    filtered = $[2]; // Берет из кеша
  }

  let returnValue;
  if ($[3] !== filtered) {
    returnValue = <List data={filtered} />;
    $[3] = returnValue;
  } else {
    returnValue = $[3];
  }

  return returnValue;
}
```

#### Почему это важно для Vite / Babel?

React Compiler — это, по сути, Babel-плагин.

- В Vite ты подключаешь его, чтобы во время трансформации исходного кода в JS для браузера, плагин успел «перехватить» твои компоненты и переписать их.
- В Next.js это уже встроено в пайплайн сборки (через Rust-компилятор SWC или тот же Babel).

### Что такое SWC (Speedy Web Compiler)?

React Compiler: Это «мозг». Он делает глубокий семантический анализ логики React. Его задача — расставить мемоизацию (useMemo/useCallback). Он про оптимизацию рендеринга. Глубоко знает React (понимает зависимости и хуки).

SWC (Speedy Web Compiler): Это «двигатель». Он заменяет Babel. Его задача — максимально быстро превратить современный TS/JSX в обычный JS. Он про скорость. Транспилятор / Минификатор (на Rust). Поверхностно знает React (просто парсит JSX).

Как они работают вместе в Next.js?
В современных версиях Next.js (15+) процесс выглядит так:

- SWC берет твой файл.
- Внутри процесса SWC запускается React Compiler (как этап трансформации).
- Компилятор анализирует, где можно воткнуть кеширование, и переписывает код.
- SWC доделывает остальную работу (превращает это в JS, понятный браузеру) и отдает результат.

#### Что отвечать на собеседовании?

Если тебя спросят: «Зачем нам React Compiler, если у нас уже есть быстрый SWC?» — отвечай так:

«Это инструменты разного назначения. SWC — это низкоуровневая инфраструктура, которая позволяет нам собирать проект за миллисекунды вместо секунд. React Compiler — это высокоуровневая надстройка над процессом сборки. Он меняет саму парадигму написания кода на React, делая оптимизацию производительности автоматической, а не ручной. Мы используем SWC как платформу, внутри которой работает React Compiler».

### Что такое «внутренний кэш» React 19?

Когда мы говорим про React Compiler, под «внутренним кэшем» подразумевается useMemoCache (это низкоуровневый хук, который не предназначен для ручного использования).

Как это работает: В обычном React при использовании useMemo(() => data, [deps]) результат хранится в объекте хука внутри Fiber-ноды. React сравнивает массив зависимостей, и если он не изменился, возвращает значение.

Внутренний кэш компилятора (React 19) работает иначе:

- Массив слотов: При сборке компилятор определяет, сколько значений в компоненте можно закешировать (переменные, JSX-узлы, функции).
- Выделение памяти: Для компонента создается фиксированный массив («кэш-фрейм»). Например, если компилятор нашел 5 сущностей для мемоизации, он создаст массив на 5 слотов.
- Прямое сравнение: Вместо того чтобы запускать логику хуков, скомпилированный код просто делает быструю проверку: if (cachedValue !== newValue) { ... }.

Важно не путать: В React 19 есть еще функция cache() (из пакета react). Она предназначена для RSC и кэширует результаты запросов к базе или API в рамках одного запроса к серверу (Request Memoization). Это совсем другая история, не связанная с ререндерами на клиенте.

### Что тако Fiber?



[Реализация структур данных](https://github.com/korzhavinilya/ts-practice/blob/master/src/data-structures/next.js/README.md).

Это сердце React, его внутренний механизм согласования (reconciliation), внедренный в версии 16.0. Для Senior-разработчика важно понимать, что Fiber — это не просто «новая версия Virtual DOM», а полная смена архитектуры с рекурсивной на итеративную, основанную на связанных списках.

Reconciliation никуда не делся. Fiber не заменил Reconciliation — Fiber заменил Stack Reconciler.

В чем разница?

- Reconciliation (Алгоритм/Правила): Это «мозги». Те самые эвристики: «Если типы не совпали — сносим всё», «Если есть key — переиспользуем». Эти правила концептуально не менялись очень давно (еще до версии 16), потому что они доказали свою эффективность для $O(n)$.
- Stack Reconciler (Старый движок): Это «старая реализация». Она выполняла алгоритм Reconciliation через рекурсию. Главный минус: она была «жадной» — если начала сравнивать дерево, её не остановить, пока стек вызовов не опустеет.
- Fiber Reconciler (Новый движок): Это «текущая реализация» (с React 16+). Она выполняет тот же алгоритм Reconciliation, но делает это итеративно (через цикл по связанному списку).
- **Тонкий момент** для Senior-собеса: Хотя сам алгоритм (правила диффинга) остался прежним, Fiber позволил добавить к нему Scheduling (планирование). Раньше Reconciliation был неразрывно связан с немедленным обновлением. Теперь между «вычислением изменений» (Reconciliation) и «отрисовкой» (Commit) появилась умная прослойка, которая может сказать: «Так, я посчитал изменения для этого списка, но сейчас юзер нажал на кнопку — я поставлю этот апдейт на паузу, обработаю клик, а потом вернусь к списку».

#### Зачем его создали? (Проблема Stack Reconciler)

До Fiber React использовал Stack Reconciler. Он работал синхронно и рекурсивно.

- Проблема: Если дерево компонентов большое, React «зависал» в глубокой рекурсии, пока не закончит всё обновление.
- Результат: Основной поток (Main Thread) блокировался, анимации лагали, а ввод в инпут подтормаживал (jank).

#### Что такое Fiber по сути?

Это единица работы (unit of work). Fiber — это обычный JS-объект, который содержит информацию о компоненте, его состоянии и связях.

Главная фишка: Fiber превратил процесс рендеринга в виртуальный «стек вызовов», которым можно управлять вручную. Теперь React может:

- Паузить работу.
- Приоритизировать задачи (например, ввод пользователя важнее фонового списка).
- Переиспользовать или выбрасывать выполненную работу.

#### Как это устроено внутри?

##### Структура данных (Linked List)

В отличие от старого Virtual DOM, который был деревом, Fiber — это связанный список. У каждого узла есть ссылки:

- child — на первый дочерний элемент.
- sibling — на соседний элемент.
- return — на родителя.
- 

Это позволяет React обходить дерево итеративно (через цикл while), а не рекурсивно. В любой момент цикл можно прервать, сохранить ссылку на текущий Fiber-узел и вернуться к нему позже.

##### Две фазы (The Two Phases)

Это ключевой момент для понимания работы useEffect и ошибок рендеринга.

Фаза 1: Render (Асинхронная). React строит Work-in-progress дерево.

- Он вызывает функции компонентов, вычисляет diff и помечает Fiber-узлы «флагами» (эффектами): Placement (вставка), Update (обновление), Deletion (удаление).
- Главное: Эту фазу можно прервать (Concurrent Mode), если прилетела более приоритетная задача (например, ввод пользователя).

Фаза 2: Commit (Синхронная). Как только дерево готово, React переходит в Commit:

- Берет «список эффектов» (готовый список изменений).
- Синхронно применяет их к реальному DOM.
- Меняет указатель: Work-in-progress дерево становится Current.
- Вызывает useLayoutEffect (синхронно) и useEffect (планирует на потом).

Процесс физической синхронизации Fiber-дерева с реальным DOM называется Commit Phase. Многие ошибочно включают его в понятие Reconciliation, но технически:

- Reconciliation — это когда мы «думаем» и вычисляем разницу.
- Commit — это когда мы «делаем» и меняем DOM.

##### Double Buffering (Двойная буферизация)

Fiber использует технику из графики: у React всегда есть два дерева.

- Current Tree: То, что сейчас отображается в DOM.
- Work-in-progress Tree: То, что React собирает прямо сейчас в фоновом режиме (черновик). Как только "черновик" готов, React просто меняет указатель, и work-in-progress становится current.

#### Приоритеты и Lanes

В React 18/19 появилась концепция Lanes (дорожки). Каждому обновлению присваивается приоритет:

- Discrete (Sync): Клик, ввод текста (нужна мгновенная реакция).
- Continuous: Скролл, драг-энд-дроп.
- Default: Загрузка данных.
- Idle: Логирование, аналитика.

Fiber позволяет React отложить «тяжелый» рендер списка (Default), если пользователь начал быстро печатать в поиске (Discrete).

#### Как это связано с твоей подготовкой к Next.js?

Именно благодаря Fiber работают Suspense и Streaming. Когда компонент «суспендится» (ждет данные), Fiber просто помечает этот узел как «не готовый», пропускает его и продолжает рендерить остальное дерево, не блокируя всё приложение.

### Reconciliation

Это «Мозги» (Render Phase). 

Декларативный алгоритм React, основанный на эвристиках, задача которого — вычислить минимально необходимое количество изменений между текущим Fiber-деревом и новым состоянием, он не бежит сразу менять DOM. Он занимается «умственным трудом»:

- Сравнивает типы узлов.
- Проверяет ключи.
- Вычисляет разницу в пропсах.
- Итог этого этапа: Список изменений (Effect List / Flags). Например: «Узел А — обновить атрибут class», «Узел Б — удалить», «Узел В — вставить».

#### Эвристический алгоритм (O(n))

Полное сравнение двух деревьев — это задача сложности $O(n^3)$. React сокращает её до $O(n)$, опираясь на два допущения:

- Разные типы — разные деревья: Если  сменился на , React не ищет сходства, а просто сносит старую ветку и строит новую.
- Keys: С помощью ключей React понимает, что элемент не изменился, а просто переместился в списке.

#### Процесс обхода (DFS по Fiber)

В отличие от старого рекурсивного обхода, Fiber использует итеративный Depth-First Search (DFS). Узлы связаны не массивом детей, а ссылками: child, sibling, return.

Алгоритм движения:

- Вниз: Идем в child, пока они есть.
- Вправо: Если детей нет, идем в sibling.
- Вверх: Если нет ни детей, ни соседей, возвращаемся в return (родитель) и ищем соседа там.

### Что такое Virtual DOM?

это концепция, а не структура данных.

Термин «Virtual DOM» — это просто паттерн, когда мы держим в памяти описание интерфейса и синхронизируем его с «реальным» DOM.

- До Fiber (Stack Reconciler): Внутри это было обычное дерево объектов. React обходил его рекурсивно.
- С Fiber: Это всё еще «виртуальное» описание UI, но теперь оно организовано как дерево из связанных списков.

#### Как «Дерево» стало «Списком»?

Fiber-узлы по-прежнему образуют иерархию (логически это дерево), но связи между ними реализованы так, чтобы их можно было обходить итеративно:

- Вместо children: [] (массив детей), у Fiber-узла есть ссылка child (только на первого ребенка).
- У этого ребенка есть ссылка sibling (на соседа).
- У каждого есть ссылка return (на родителя).

Почему это важно? Рекурсию нельзя поставить на паузу. Если ты зашел в функцию, ты должен из неё выйти. Итерацию (цикл while) — можно. Именно структура связанного списка позволила React реализовать конкурентный рендеринг: остановиться посередине списка, дать браузеру отрисовать кадр (анимацию) и вернуться к работе.

#### Значит ли это, что Virtual DOM «умер»?

В официальной документации React термин «Virtual DOM» сейчас встречается редко. Команда React предпочитает говорить о «Fiber-дереве».

- Virtual DOM часто ассоциируется с «диффингом» (сравнением двух огромных JSON-объектов).
- Fiber — это про «планирование» (scheduling) и работу с приоритетами.

### Что такое Concurrent React?

Concurrent React принес нам возможность делать интерфейсы, которые не лагают, даже когда React занят тяжелыми вычислениями. Для Senior-разработчика это переход от «ручного» дебаунсинга (setTimeout) к декларативному управлению приоритетами.

Главные герои здесь: useTransition и useDeferredValue.

Они позволяют нам уйти от блокирующего рендеринга. Мы можем разделять UI на критические и некритические части. React управляет этим на уровне Fiber-движка: он может прерывать низкоприоритетную работу ради высокоприоритетной. Это заменяет устаревшие паттерны типа дебаунсинга более естественным поведением, которое адаптируется под мощность устройства пользователя.

Если тебе нужно не частить запросами на бэкенд (Search API) — используй Debounce.
Если тебе нужно не фризить UI при тяжелых локальных вычислениях или больших списках — используй Concurrent Hooks.

Пример из жизни: Ты пишешь поиск. Юзер вводит текст. Ты используешь Debounce для fetch('/api/search'), чтобы не дергать базу на каждой букве. Но результат этого фетча (массив из 1000 строк) ты прокидываешь через useDeferredValue, чтобы при отрисовке этого массива страница не превратилась в тыкву.

- `useTransition` — «Плавное переключение»
  Этот хук позволяет пометить обновление стейта как «неприоритетное». React начнет рендерить его в фоне, но если юзер что-то нажмет или введет в это время, React бросит фоновую работу и сначала обработает ввод.
  Пример: Поиск по огромному списку Представь, что при каждом вводе в инпут ты фильтруешь 5000 элементов. Без useTransition инпут будет «лагать».
  Что происходит под капотом: React создает Work-in-progress дерево для filteredItems. Если ты быстро печатаешь, React видит, что прилетают новые setFilter, и просто выбрасывает старую незавершенную работу по фильтрации, начиная новую. Инпут при этом остается «резиновым» и отзывчивым.
- `useDeferredValue` — «Отложенное значение»
  Работает похоже на useTransition, но когда у тебя нет доступа к функции setState (например, значение прилетает сверху как пропс).
  В чем разница с Debounce?
  - Debounce: Ждет фиксированное время (например, 300мс), прежде чем начать работу. Это всегда искусственная задержка.
  - DeferredValue: Начинает работу сразу, как только процессор освободился. На мощных девайсах задержки не будет вообще, на слабых — она будет ровно такой, какая нужна.
- `useActionState` (React 19 / Next.js 15)
  Это новый стандарт для работы с Server Actions. Он заменяет старый useFormState и автоматически использует переходы (transitions) под капотом.
  - Автоматический isPending без лишних ререндеров. 
    Раньше, чтобы сделать лоадер, мы вручную писали setIsLoading(true), ждали fetch, а потом setIsLoading(false).
    Как только ты нажал на кнопку, React помечает всё последующее обновление как Transition (низкий приоритет).
    Если в процессе этой «загрузки» на странице произойдет что-то более важное (например, анимация или ввод в другой инпут), React отдаст приоритет им, не блокируя поток ожиданием ответа от сервера.
  - Ожидание "Свежих данных" (RSC Payload)
    Поскольку это Transition, React на клиенте не будет «дёргать» UI, пока не получит этот новый RSC Payload полностью. Он держит старую страницу интерактивной, пока «в фоне» готовит новую версию страницы.
    !TIP Без транзакции (если бы ты делал это через обычный useEffect и fetch), тебе пришлось бы самому думать, как синхронизировать момент «конец загрузки» и «момент, когда новые данные реально отрисовались». useActionState делает это атомарно.

### Каверзные вопросы для Senior по Next.js 15 / React 19

#### В чем разница между Server Components и SSR?

SSR — это способ генерации HTML. RSC — это тип компонента. RSC никогда не ререндерятся на клиенте и могут передавать данные в Client Components без сериализации в JSON.

#### Что произойдет, если вызвать Server Action из обычного  в клиентском компоненте?

Это сработает. Next.js сделает POST-запрос. Но ты потеряешь преимущества Progressive Enhancement (форма не сработает без JS). Также тебе придется вручную обрабатывать лоадеры через useTransition.

#### Почему RSC Payload содержит ссылки на клиентские компоненты, а не их код?

Чтобы минимизировать размер ответа. Клиентские компоненты уже находятся в JS-бандлах. RSC Payload просто говорит React: «В этом месте дерева вставь компонент из файла X с пропсами Y».

#### Как useOptimistic понимает, когда нужно откатить состояние?

Он жестко связан с жизненным циклом Server Action. Как только промис экшена завершается (успешно или с ошибкой), useOptimistic сбрасывает временное состояние, а страница обновляется реальными данными от revalidatePath.

#### Можем ли мы использовать Context API внутри Server Components?

Нет. Context — это рантайм-инструмент для передачи стейта в дереве, которое постоянно живет и меняется. RSC рендерятся один раз на сервере. Для передачи данных между RSC используются обычные props или общие модули данных (кэшированные функции).

#### RSC vs Client Components: Где граница сериализации?

Суть: Когда Server Component рендерит Client Component, данные (props) передаются через RSC Payload.

- Что можно передавать: Только сериализуемые данные. Простые объекты, массивы, строки, числа, булевы значения и итерируемые объекты (которые React умеет парсить).
- Что нельзя передавать: Функции (события типа onClick), сложные классы с методами, или специфические браузерные API.
- Почему так? Функция — это живой код в памяти Node.js. Ты не можешь «переслать» живую функцию по HTTP в браузер. Браузер должен получить только «описание» данных, чтобы воссоздать их у себя.

Senior-нюанс: Ты можешь передать Server Action в клиентский компонент как пропс. React под капотом заменит его на специальную ссылку (ID), и когда клиент вызовет эту «функцию», он на самом деле сделает сетевой запрос к серверу.

#### Hydration: Почему возникает ошибка "Text content did not match"?

Суть: Гидрация — это процесс, когда React в браузере пытается «наложить» свою виртуальную структуру на HTML, пришедший с сервера. Если они не совпадают — всё ломается.

Типичные причины:

- Недетерминированные данные: Использование new Date() или Math.random() прямо в теле компонента без useEffect. На сервере было «10:00:01», а пока пакет долетел, на клиенте уже «10:00:02».
- Браузерные API: Использование window или localStorage при первом рендере. На сервере этого нет (будет пустая строка/undefined), а на клиенте есть.
- Расширения (Extensions): Иногда плагины браузера (переводчики, менеджеры паролей) вклеивают свой HTML в твой код, сбивая React с толку.

Как лечить:

- Переносить работу с датами/окном в useEffect (он запускается только на клиенте).
- Использовать хук useSyncExternalStore для внешних данных.
- В крайнем случае — пропс suppressHydrationWarning (но это костыль).

#### Caching: Разница между cache(), fetch и revalidate

В Next.js 15 есть четыре уровня кэширования. Важно их не путать:


| Инструмент      | Уровень             | Что делает?                                                                                                                                  |
| --------------- | ------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| cache() (React) | Request Memoization | Если ты в 5 разных RSC вызываешь одну и ту же функцию getUser(), запрос в базу уйдет только один раз за время обработки одного HTTP-запроса. |
| fetch (Next.js) | Data Cache          | Сохраняет результат запроса к API между разными пользователями и запросами (на диске сервера).                                               |
| revalidatePath  | Invalidating        | Команда серверу: «Данные изменились, удали старый кэш для этого URL и пересобери страницу».                                                  |
| Router Cache    | Client-side         | Кэш в браузере. Когда ты ходишь по , Next хранит уже посещенные страницы в памяти, чтобы не запрашивать их снова.                            |


#### Streaming: Как Suspense влияет на TTFB?

TTFB (Time To First Byte) — это время от нажатия Enter до получения первого байта данных от сервера.

- Без Streaming: Сервер ждет все данные -> рендерит весь HTML -> отправляет ответ. Если данные грузятся 3 секунды, пользователь 3 секунды видит белый экран. TTFB = 3с.
- Со Streaming: Сервер отправляет head и layout мгновенно. Соединение остается открытым. TTFB = 50мс. Пока пользователь видит скелетон, тяжелые данные «доплывают» по тому же каналу.

Senior-совет: Упомяни, что стриминг полезен не только для UX, но и для SEO. Роботы Google получают первый байт быстрее, что положительно влияет на Core Web Vitals (в частности, на LCP и TTFB).

#### Fiber & Reconciliation: Почему это позволяет делать useTransition?

Это вопрос на «миллион».

- Раньше (Stack Reconciler) рендеринг был «атомарным»: ты не мог его прервать.
- В Fiber обход итеративный. Каждый узел — это «Unit of Work».
- useTransition говорит React: «Это обновление (например, фильтрация огромного списка) имеет низкий приоритет».
- React начинает работать над этим обновлением в фоновом Fiber-дереве. Если в этот момент пользователь что-то вводит в инпут (высокий приоритет), Fiber-движок приостанавливает фоновую работу, обрабатывает инпут, отрисовывает его, и только в свободное время возвращается к фильтрации списка.
